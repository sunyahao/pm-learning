# 仓库结构规范

## 目的

定义仓库的 OpenSpec 标准结构，以及规范层和派生学习资产之间的边界。

## 要求

### 要求：仓库必须具备标准 OpenSpec 结构
仓库必须暴露标准 OpenSpec 结构，至少包括根目录 `AGENTS.md`、`openspec/config.yaml`、`openspec/project.md`、`openspec/specs/` 和 `openspec/changes/`。

#### 场景：检查仓库根目录
- **前提** 贡献者打开仓库根目录
- **当** 他查看顶层文件和目录
- **则** 能看到根目录的 `AGENTS.md`
- **并且** 能看到 `openspec/config.yaml`
- **并且** 能看到 `openspec/project.md`
- **并且** 能看到 `openspec/specs/`
- **并且** 能看到 `openspec/changes/`

### 要求：必须区分规范源头和派生资产
仓库必须将 `openspec/specs/` 视为规范源头，并将 `chapters/` 视为遵循规范生成或维护的派生学习资产。

#### 场景：判断哪个目录是源头
- **前提** 贡献者需要确认仓库行为由谁定义
- **当** 他比较 `openspec/specs/` 与 `chapters/`
- **则** 能确认 `openspec/specs/` 是规范源头
- **并且** 能确认 `chapters/` 是派生输出层

### 要求：仓库可以包含总览型备考指南目录
仓库可以在根目录下维护 `guides/` 目录，用于保存跨章节的备考重点、趋势分析和冲刺计划。

#### 场景：查找总览型备考资料
- **前提** 读者需要查看跨章节的备考策略
- **当** 他查看仓库根目录
- **则** 可以在 `guides/` 中找到总览型策略文件

### 要求：总览型导学可以串联多个章节
`guides/` 可以包含基于 24 章目录的串联导学、学习地图或阅读路线，用于解释章节之间的依赖关系、学习顺序和复习优先级，但不得替代 `chapters/` 的一章一目录映射。

#### 场景：读者想先建立全书整体理解
- **前提** 读者觉得单章阅读难以形成全局框架
- **当** 他查看 `guides/`
- **则** 可以找到帮助串联多个章节的总览型导学
- **并且** 导学会继续指向对应的章节目录，而不是替代章节目录

### 要求：guides 可以提供资料总览与提分导航
`guides/` 可以包含面向备考通过目标的资料盘点、来源导航和优化建议文档，用于说明主来源、真题来源、辅助资料、当前资产覆盖情况以及下一步复习优先级。

#### 场景：读者想先知道仓库里什么资料最值得看
- **前提** 读者面对多个资料目录，不知道先用哪类资料
- **当** 他查看 `guides/`
- **则** 可以找到按“章节主线、论文、案例、选择题、冲刺资料”整理的总览文档
- **并且** 能据此判断当前最值得投入时间的学习材料

### 要求：归档的结构变更必须保留
仓库必须在 `openspec/changes/archive/` 下保留已完成的结构变更记录。

#### 场景：追溯当前结构为何形成
- **前提** 贡献者想知道仓库为何采用当前结构
- **当** 他查看 `openspec/changes/archive/`
- **则** 能找到包含 proposal、design、tasks 和规范增量的已归档变更

### 要求：OpenSpec 规范文档必须使用中文
位于 `openspec/` 下的规范性文档必须使用中文编写；如需保留英文能力标识，应仅限于目录名或文件名，不得影响正文语言。

#### 场景：检查规范语言
- **前提** 贡献者查看 `openspec/` 下的规范文档
- **当** 他阅读文档正文
- **则** 正文内容使用中文表达
- **并且** 中文要求已被规范本身明确约束
