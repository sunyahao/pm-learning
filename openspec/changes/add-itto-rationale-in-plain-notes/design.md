# 设计：白话拆解中的 ITTO 成因解释

## 设计目标

白话拆解文档本来就承担“把抽象概念讲活”的职责。  
这次新增的不是一份新文件，而是在白话拆解里增加一个固定能力：

`解释为什么这个过程会吃进这些输入、用这些工具、产出这些输出。`

## 设计原则

### 1. 不做 ITTO 列表重复

`ITTO输出速看.md` 已经负责快速列出关键输出物。  
白话拆解新增的部分，不应该再机械抄一遍列表，而是解释：

- 这个过程到底在干什么
- 既然它做的是这件事，那为什么自然会需要这些输入
- 既然它做的是这件事，那为什么自然会用这些工具与技术
- 既然它做的是这件事，那为什么自然会产出这些输出
- 这些关键输入、关键工具和输出在项目里分别扮演什么角色

### 2. 优先解释关键 ITTO，至少覆盖关键输入、关键工具与关键输出

为了避免篇幅失控，不要求把整张 ITTO 表全部讲完。  
但样板阶段至少应解释关键输入、关键工具与关键输出的逻辑来源。

### 3. 结构应贴近过程顺序

推荐结构：

1. 先说这个过程的本质动作
2. 再解释为什么会自然需要某几个关键输入
3. 再解释为什么会自然使用某几个关键工具
4. 再解释为什么会自然产生某几个关键输出
5. 最后补一句“这就是为什么题里会考到这些 ITTO”

## 第12章样板范围

第12章 `质量管理` 的白话拆解应增加一段“为什么是这些 ITTO”，至少覆盖：

- `规划质量管理`
- `管理质量`
- `控制质量`

并解释各自关键输入、关键工具与关键输出背后的逻辑。
