# 设计：白话拆解文档

## 设计目标

`白话拆解.md` 不是替代 `精讲.md` 或 `过程场景讲解.md`，而是补它们之间的一个理解层：

- `精讲.md` 负责系统讲
- `过程场景讲解.md` 负责放回项目过程里讲
- `白话拆解.md` 负责把最难啃、最容易“看定义像懂了，实际没懂”的术语讲明白

## 文档定位

适用场景：

- 某个章节里存在高频术语、关系逻辑、方法对比，学习者容易死记硬背
- 这些概念仅靠正式定义不够，需要补“大白话 + 业务例子 + 记忆抓手”

不适用场景：

- 纯粹重复 `完整整理.md`
- 与原 PDF 无关的自由发挥
- 把整章内容重新写一遍，导致和 `精讲.md` 职责重叠

## 推荐结构

一份 `白话拆解.md` 应优先采用下面结构：

1. 这份文档解决什么问题
2. 最容易发虚的术语有哪些
3. 每个术语的四层解释
   - 标准说法
   - 大白话
   - 公司项目里的例子
   - 一句话记忆
4. 最容易混淆的对比
5. 放到同一个项目链路里怎么看
6. 考试里怎么用

## 第10章样板范围

第10章 `进度管理` 的 `白话拆解.md` 应覆盖：

- `FS`、`SS`、`FF`、`SF`
- `PDM`、`ADM`
- `提前量`、`滞后量`
- `里程碑`
- `关键路径`
- `赶工`、`快速跟进`
- `进度基准` 与 `项目进度计划`

## 学习入口顺序

如章节存在 `白话拆解.md`，学习入口顺序建议为：

1. `完整整理.md`
2. `精讲.md`
3. `过程场景讲解.md`
4. `白话拆解.md`
5. `复盘题单.md`
6. `参考作答.md`

这样可以先建立正式理解，再用口语化方式把难点打通，最后进入主动回忆。
