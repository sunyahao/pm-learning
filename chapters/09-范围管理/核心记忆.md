<a id="core-mainline"></a>
## 一、整章主线

- 范围管理先回答“做什么”，再回答“怎么拆”，最后回答“怎么验”和“怎么控”。
- 最核心链条：`需求 -> 项目范围说明书 -> WBS -> 验收 -> 范围变更控制`
- 六过程口诀：`规 -> 收 -> 定 -> 创 -> 确 -> 控`
- 这章最怕丢边界，一旦边界没立住，后面进度、成本、变更都会连着出问题。

<a id="core-basics"></a>
## 二、管理基础

### 1. 最该背

- 产品范围回答“产品要具备什么功能和特性”。
- 项目范围回答“为了交付这些成果，项目团队要做哪些工作”。
- 范围管理的起点不是写 WBS，而是先把需求收清楚。
- 范围基准 = `项目范围说明书 + WBS + WBS词典`

### 2. 高频文件

- `需求文件` 偏记录需求本身。
- `需求跟踪矩阵` 偏把需求从提出到交付一路跟踪到底。
- `项目范围说明书` 偏边界、可交付成果、验收标准和除外责任。

## 三、六过程

### 1. 规划范围管理

- 先把规则定下来：范围怎么定义、怎么分解、怎么确认、怎么控制。
- 高频输出：`范围管理计划 + 需求管理计划`

### 2. 收集需求

- 这一过程是在把需求“收上来、说清楚、记下来”。
- 高频输出：`需求文件 + 需求跟踪矩阵`

展开阅读：

### 3. 定义范围

- 这一步真正把项目边界写清楚。
- 高频输出：`项目范围说明书`

展开阅读：
- [完整整理.md](./完整整理.md)
- [项目范围说明书.md](./输出物/03-定义范围/项目范围说明书.md)

### 4. 创建 WBS

- WBS 是把范围逐层拆成可管理的工作结构。
- 最该记：`范围基准` 在这一步形成。
- 高频原则：`100% 原则`

展开阅读：
- [完整整理.md](./完整整理.md)
- [WBS.md](./输出物/04-创建WBS/WBS.md)
- [WBS词典.md](./输出物/04-创建WBS/WBS词典.md)

### 5. 确认范围

- 关键词：`正式验收`
- 这一步不是团队自己说“做完了”，而是客户或发起人正式接受成果。
- 高频输出：验收后的可交付成果、工作绩效信息、变更请求。

展开阅读：
- [完整整理.md](./完整整理.md)
- [验收确认单.md](./输出物/05-确认范围/验收确认单.md)

### 6. 控制范围

- 关键词：`维护范围基准`
- 这一步负责防范围蔓延，发现偏差后走正式变更。
- 高频输出：工作绩效信息、变更请求。

<a id="core-confusions"></a>
## 四、最易混点

- 产品范围 vs 项目范围
  一个偏成果特性，一个偏项目工作。
- 需求文件 vs 需求跟踪矩阵
  一个偏把需求写清楚，一个偏把需求跟到底。
- 确认范围 vs 控制范围
  一个偏正式验收，一个偏维护范围基准。
- 确认范围 vs 控制质量
  一个问“客户收不收”，一个问“成果合不合格”。
- 范围蔓延 vs 正常变更
  一个是未经批准的扩张，一个是走正式控制流程的受控变化。

<a id="core-review"></a>
## 五、案例题抓手

- 如果题目说“客户不断顺手加需求”，先想到需求没管住、边界没立住、变更没受控。
- 如果题目说“后期频繁返工”，先顺着这条线排查：`需求 -> 范围说明书 -> WBS -> 验收`
- 如果题目问“为什么客户不签字”，先区分是确认范围问题还是控制质量问题。
- 范围题很爱和整合管理、进度管理一起联动考。
