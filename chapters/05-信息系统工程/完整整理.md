## 一、软件架构风格是本章第一大块

高频架构风格通常包括：

- 数据流风格
- 调用/返回风格
- 独立构件风格
- 虚拟机风格
- 仓库风格

复习时不要只背名字，要配合“代表模式和适用思路”一起记。

### 1. 数据流风格

更偏：

- 数据在多个处理步骤之间按顺序流动
- 常见感受是“输入经过一层层处理后输出”

常见样例：

- 编译器把源代码依次经过词法分析、语法分析、语义分析、目标代码生成
- ETL 处理把原始数据依次经过抽取、清洗、转换、装载后进入数据仓库
- 日志处理流水线把采集到的日志依次做过滤、聚合、告警分析

代表模式和记忆抓手：

- 批处理序列：一批数据做完一步，再进入下一步
- 管道-过滤器：数据像在管道里流动，每个过滤器负责一段处理

考试里看到“数据一层层加工后输出”，通常就要往数据流风格上想。

### 2. 调用/返回风格

更偏：

- 模块之间通过调用和返回完成协作
- 更容易联想到过程式、分层式等常见组织方式

常见样例：

- 一个典型管理信息系统中，界面层调用业务层，业务层再调用数据访问层
- 一个订单系统中，订单模块调用库存模块、支付模块，再把结果返回给上层
- 传统过程式程序里，主程序调用多个子程序来完成整体业务

代表模式和记忆抓手：

- 主程序/子程序
- 面向对象系统
- 分层架构

考试里如果强调“上层调下层、模块之间一层层调用”，往往更接近调用/返回风格。

### 3. 独立构件风格

更偏：

- 系统由相对独立的组件组成
- 组件之间通过约定方式协作

常见样例：

- 电商系统里订单服务、库存服务、支付服务、通知服务分别独立部署
- 一个事件驱动系统中，前台下单后发布事件，库存、积分、短信等组件分别订阅处理
- 操作系统或平台把不同插件、组件按标准接口接入

代表模式和记忆抓手：

- 事件系统
- 消息总线协作
- 组件化或服务化部署

它最像“每个部件先独立干自己的事，再按规则协同”。

### 4. 虚拟机风格

更偏：

- 通过解释执行、规则执行或虚拟环境支撑系统运行

常见样例：

- Java 程序运行在 JVM 上，由虚拟机解释或执行字节码
- SQL 语句由数据库解释执行
- 工作流引擎、规则引擎根据预定义规则去解释并驱动业务动作

代表模式和记忆抓手：

- 解释器
- 规则系统
- 虚拟执行环境

看到“不是直接硬编码执行，而是先交给一个解释或运行环境处理”，就可以联想到虚拟机风格。

### 5. 仓库风格

更偏：

- 围绕共享数据仓库或中心数据结构组织系统

常见样例：

- 多个业务模块都围绕同一数据库读写数据
- 数据分析平台围绕中心数据仓库组织报表、挖掘和分析应用
- 黑板系统中，不同求解模块围绕共享知识库协同工作

代表模式和记忆抓手：

- 数据库系统
- 黑板模型
- 共享仓库或共享知识库

这类风格最像“大家围着同一个数据中心干活”。

### 6. 架构风格为什么总被考

因为它不是几个术语堆在一起，而是在提醒你：

- 系统怎么分
- 模块怎么协作
- 数据怎么流
- 能力怎么复用

也就是说，很多题目其实不会直接问你“这叫什么风格”，而是会先给你一个样例：

- 是不是把数据按流水线一步步处理
- 是不是上层模块不断调用下层模块
- 是不是多个独立部件靠消息或事件协作
- 是不是依赖解释器、规则引擎或虚拟运行环境
- 是不是围绕中心仓库或共享数据库组织系统

你只要先识别出样例背后的组织方式，风格就更容易选对。

所以做题时要先判断题目在描述哪种“系统组织思路”，而不是只盯术语名字。

## 二、需求工程要看“全过程”

这是本章特别高频的一块。

常见主线包括：

- 需求获取
- 需求分析
- 需求规格说明
- 需求验证与确认

考试最常卡的地方是：

- 需求验证和需求确认分不清
- 把需求过程写成一次性动作，不理解它是持续细化的工程活动

### 1. 需求获取

重点在于：

- 把用户、干系人的真实需求收集上来
- 弄清问题背景、使用场景和目标

### 2. 需求分析

重点在于：

- 整理、归纳、筛选需求
- 发现冲突、模糊和边界问题

### 3. 需求规格说明

重点在于：

- 把需求写成清晰、正式、可沟通、可实现的文档

### 4. 需求验证与确认

重点在于：

- 验证：更偏“需求描述本身对不对”
- 确认：更偏“这是不是用户真正想要的”

这是本章最容易混的一对概念。

一句话记忆：

- 验证：`写得对不对`
- 确认：`要得对不对`

## 三、QFD 和需求层次要会辨析

本章不仅讲需求过程，也会强调：

- 用户想要什么
- 系统应该提供什么
- 需求如何逐步转换成可实现的工程描述

QFD（Quality Function Deployment，质量功能展开）可以理解为：

- 一种把用户需求系统化地转换为产品或系统技术要求的方法
- 它强调先听清用户真正关心什么
- 再把这些需求逐步分解、映射成设计、功能、部件和实现要求

QFD 这类内容的价值就在于把“用户之声”和“工程实现”建立联系。

更白话一点：

- 它不是只管记录需求
- 而是帮助把用户语言转成工程语言

### 1. QFD 到底在解决什么问题

很多项目失败，不是因为团队不会开发，而是因为：

- 用户说的是业务感受
- 开发听到的是技术任务
- 中间缺了一层清晰的转换

QFD 想解决的，就是这层“需求翻译”问题。

它关心的是：

- 用户最在意什么
- 这些在意点应该对应哪些系统功能或质量特性
- 这些功能再往下应该落实成哪些设计和实现要求

### 2. QFD 的核心思想怎么记

最稳的记法不是背流程图，而是记住这条线：

- 先收集用户之声
- 再识别关键需求
- 再把需求转换成技术特性
- 最后把技术特性继续分解到设计与实现

一句话记忆：

- `QFD 就是把“用户想要什么”一步步翻译成“系统该怎么做”。`

### 3. 为什么它和需求工程放在一起

因为需求工程解决的是“把需求收集、分析、写清、验证和确认”，而 QFD 更强调：

- 怎样把用户需求继续往工程实现方向推进
- 怎样让需求、设计和实现之间建立对应关系

所以它不是脱离需求工程的独立知识点，而是需求向设计转换时的重要抓手。

### 4. 可以用一个很直观的例子来理解

假设用户说：

- 系统要好用
- 查询要快
- 下单不要出错

这些说法本身还是偏业务语言、感受语言。

用 QFD 的思路，就会继续追问并转换成更可实现的要求，比如：

- “好用”可以进一步转成界面简洁、关键功能步骤少、常用操作入口清楚
- “查询要快”可以进一步转成响应时间指标、索引优化、缓存设计
- “下单不要出错”可以进一步转成订单校验规则、库存校验机制、异常提示机制

这样，团队拿到的就不再只是模糊愿望，而是更接近设计和开发的工程要求。

所以考试如果问 QFD，重点不是展开复杂步骤，而是记住它的桥梁作用：

- 把用户需求往工程实现方向转化
- 把业务语言逐步翻译成技术语言
- 让需求、设计和实现之间建立可追踪关系

## 四、模型相关内容要看各自表达什么

常见高频抓手包括：

- 数据模型
- 功能模型
- 行为模型

做题时最稳的办法是问自己：

- 这个模型更偏描述对象结构
- 还是描述系统做什么
- 或者描述系统如何随时间变化

### 1. 数据模型

更偏：

- 实体
- 属性
- 关系

重点在于把对象结构说清。

### 2. 功能模型

更偏：

- 系统要完成什么功能
- 输入输出如何流转

重点在于把“系统做什么”说清。

### 3. 行为模型

更偏：

- 状态变化
- 动态过程
- 事件触发

重点在于把“系统怎么变化”说清。

一句话记忆：

- 数据模型看结构
- 功能模型看功能
- 行为模型看动态

## 五、安全工程是本章另一条主线

安全工程常见高频抓手包括：

- 三维空间理解
- 安全服务
- 安全能力成熟度相关概念

复习时不要只把它看成“第02章安全基础的重复”，它更偏：

- 用工程化视角看安全
- 把安全纳入系统开发和建设过程

### 1. 为什么安全工程不是“最后补一下”

因为安全如果不在设计和开发阶段纳入，后期往往会出现：

- 大量返工
- 架构调整成本高
- 权限和接口问题集中暴露

所以安全工程的意义在于：

- 尽早把安全要求嵌入系统建设全过程

### 2. 安全服务怎么看

这类题通常不是展开特别细的技术，而是看你是否知道：

- 安全不是一个孤立动作
- 它包含多种服务和保障能力

### 3. 安全能力成熟度怎么看

题目更想考你的是：

- 安全能力也可以分层次、分成熟度来建设
- 不是“有安全”或“没安全”这么简单
