## 这章在考什么

<a id="integration-position"></a>

整合管理不是“再多一个知识领域”，而是项目经理如何把所有知识领域拉到同一套管理节奏里。考试很喜欢用它来统领范围、进度、成本、变更、收尾和项目文件。

<a id="integration-basics"></a>
## 一、管理基础

<a id="integration-goals"></a>
### 1. 项目整合管理的目标

- 资源分配
- 平衡竞争性需求
- 研究各种备选方法
- 裁剪过程以实现项目目标
- 管理各个项目管理知识领域之间的依赖关系

### 2. 整合管理的责任主体

- 项目整合管理由项目经理负责。
- 项目经理要整合所有其他知识领域的成果，并掌握项目总体情况。
- 整合管理责任不能被授权或转移，项目经理必须对整个项目承担最终责任。

<a id="integration-layers"></a>
### 3. 执行整合的三个层面

- 过程层面：有些过程只发生一次，但很多过程在项目期间会重叠并反复出现。需求变化会联动范围、进度、预算，并进一步触发变更请求。
- 认知层面：项目经理的人际关系技能、管理风格和方法选择，取决于项目规模、复杂性和组织文化。
- 背景层面：新技术、社交网络、多元文化、虚拟团队和新价值观不断出现，项目经理要结合新的环境因素执行整合。

<a id="integration-complexity"></a>
### 4. 项目的复杂性

项目复杂性至少有四个典型特征：

- 包含多个部分
- 不同部分之间存在一系列关联
- 不同部分之间存在动态交互作用
- 这些交互作用形成的行为远大于各部分简单相加

<a id="integration-plan-vs-files"></a>
### 5. 项目管理计划与项目文件的区别

项目管理计划更偏“怎么管”，项目文件更偏“管理过程中形成了哪些记录和支撑材料”。

项目管理计划通常包括：

- 各子管理计划：范围、需求、进度、成本、质量、资源、沟通、风险、采购、干系人、变更、配置
- 各基准：范围基准、进度基准、成本基准
- 绩效测量基准、项目生命周期描述、开发方法等

项目文件通常包括：

- 活动属性、活动清单、假设日志、估算依据、变更日志、成本估算、持续时间估算
- 问题日志、经验教训登记册、里程碑清单、项目日历、项目沟通记录、项目进度计划、项目进度网络图
- 项目范围说明书、项目团队派工单、质量控制测量结果、质量测量指标、质量报告
- 需求文件、需求跟踪矩阵、资源分解结构、资源日历、资源需求、风险登记册、风险报告
- 进度数据、进度预测、干系人登记册、团队章程、测试与评估文件

### 6. 整合管理的裁剪考虑因素

整合管理在裁剪时，通常会关注：

- 项目生命周期
- 开发生命周期
- 管理方法
- 知识管理
- 变更
- 治理
- 经验教训
- 效益

## 二、过程总览

<a id="integration-processes"></a>

整合管理共 7 个过程：

1. 制定项目章程
2. 制订项目管理计划
3. 指导与管理项目工作
4. 管理项目知识
5. 监控项目工作
6. 实施整体变更控制
7. 结束项目或阶段

它们可以理解为一条完整闭环：

启动授权 -> 制定总计划 -> 推动执行 -> 沉淀知识 -> 监控绩效 -> 受控变更 -> 正式收尾

## 三、过程逐项整理

<a id="integration-charter"></a>
### 1. 制定项目章程

定义：
编写一份正式批准项目并授权项目经理在项目活动中使用组织资源的文件的过程。

作用：

- 明确项目与组织战略目标之间的直接联系
- 确立项目的正式地位
- 展示组织对项目的承诺

频次：
仅开展一次，或者仅在项目的预定义时点开展。

主要输入：

- 立项管理文件，如商业论证、成本效益分析
- 协议
- 事业环境因素
- 组织过程资产

常见工具与技术：

- 专家判断
- 数据收集：头脑风暴、焦点小组、访谈
- 人际关系与团队技能：冲突管理、引导
- 会议

主要输出：

- 项目章程
- 假设日志

高频原文点：

- 项目章程授权项目经理开展规划、执行和控制，并授权项目经理使用组织资源。
- 项目经理应尽量早任命，最好在制定项目章程时就确认。
- 项目章程可由发起人编制，也可由项目经理与发起机构合作编制。
- 章程一经批准，就标志项目正式启动。
- 启动项目的通常是项目外部机构，如发起人、PMO（Project Management Office，项目管理办公室）、项目组合治理委员会主席或其授权代表。

项目章程通常记录的高层级内容包括：

- 项目目的
- 可测量的项目目标和成功标准
- 高层级需求、高层级项目描述、边界与主要可交付成果
- 整体项目风险
- 总体里程碑进度计划
- 预先批准的财务资源
- 关键干系人名单
- 项目审批要求和退出标准
- 项目经理的职责与职权
- 发起人或批准章程人员的姓名与职权

<a id="integration-plan"></a>
### 2. 制订项目管理计划

定义：
定义、准备和协调项目计划的所有组成部分，并把它们整合为一份综合项目管理计划的过程。

作用：
形成一份综合文件，用于确定所有项目工作的基础及执行方式。

主要输入：

- 项目章程
- 其他规划过程的输出
- 事业环境因素
- 组织过程资产

工具与技术：

- 专家判断
- 数据收集：头脑风暴、核对单、焦点小组、访谈
- 人际关系与团队技能：冲突管理、引导
- 会议

主要输出：

- 项目管理计划

高频原文点：

- 项目管理计划必须基准化，至少应规定范围、时间、成本三大基准。
- 在基准确定之前，可多次更新项目管理计划，通常不必走正式流程。
- 一旦基准确定，后续更新必须通过变更请求和整体变更控制过程。
- 在项目收尾前，项目管理计划会持续渐进明细，但更新需要受控和批准。

项目管理计划的组成通常包括：

- 子管理计划：范围、需求、进度、成本、质量、资源、沟通、风险、采购、干系人参与
- 基准：范围基准、进度基准、成本基准
- 其他组件：变更管理计划、配置管理计划、绩效测量基准、项目生命周期、开发方法、管理审查

<a id="integration-direct"></a>
### 3. 指导与管理项目工作

定义：
为实现项目目标而领导和执行项目管理计划中所确定的工作，并实施已批准变更的过程。

作用：
对项目工作和可交付成果开展综合管理，提高项目成功可能性。

主要输入：

- 项目管理计划
- 项目文件
- 批准的变更请求
- 事业环境因素
- 组织过程资产

工具与技术：

- 专家判断
- 项目管理信息系统
- 会议

主要输出：

- 可交付成果
- 工作绩效数据
- 问题日志
- 变更请求
- 项目管理计划更新
- 项目文件更新
- 组织过程资产更新

高频原文点：

- 批准的变更请求通常来自整体变更控制过程。
- CCB（Change Control Board，变更控制委员会）负责对变更做决策，是决策机构，不是作业机构。
- CCB 通过评审决定是否需要变更项目基准，但不负责提出变更方案。
- 常见变更请求包括纠正措施、预防措施、缺陷补救和更新。
- 工作绩效数据包括已完成工作、KPI（Key Performance Indicator，关键绩效指标）、技术绩效测量、实际开始与完成日期、已完成故事点、变更请求数量、缺陷数量、实际成本和实际持续时间等。

<a id="integration-knowledge"></a>
### 4. 管理项目知识

定义：
使用现有知识并生成新知识，以实现项目目标并帮助组织学习的过程。

作用：

- 利用已有组织知识创造或改进项目成果
- 使当前项目创造的知识可用于支持组织运营和未来项目

主要输入：

- 项目管理计划
- 项目文件
- 可交付成果
- 事业环境因素
- 组织过程资产

工具与技术：

- 专家判断
- 知识管理
- 信息管理
- 人际关系与团队技能
- 会议

主要输出：

- 经验教训登记册
- 项目管理计划更新
- 组织过程资产更新

高频原文点：

- 知识管理偏向分享隐性知识，如实践社区、工作跟随、专题研讨、讲故事、知识分享活动等。
- 信息管理偏向分享显性知识，如经验教训登记册、图书馆服务、信息收集和项目管理信息系统。
- 经验教训登记册在项目早期就应创建，并在项目期间不断更新。
- 经验教训可记录挑战、问题、已识别风险、机会以及建议和行动方案。

<a id="integration-monitor"></a>
### 5. 监控项目工作

定义：
跟踪、审查和报告整体项目进展，以实现项目管理计划中确定的绩效目标的过程。

作用：

- 让干系人了解项目当前状态
- 通过成本和进度预测帮助干系人了解未来状态

主要输入：

- 项目管理计划
- 项目文件
- 工作绩效信息
- 协议
- 事业环境因素
- 组织过程资产

常见工具与技术：

- 数据分析：备选方案分析、成本效益分析、挣值分析、根本原因分析、趋势分析、偏差分析
- 决策
- 会议

主要输出：

- 工作绩效报告
- 变更请求
- 项目管理计划更新
- 项目文件更新

高频原文点：

- 监督偏重“发现问题”，控制偏重“解决问题”。
- 工作绩效报告通常通过沟通过程发送给干系人。
- 常见表现形式包括状态报告、进展报告、挣值图表、趋势线、预测、储备燃尽图、缺陷直方图、仪表板和信号灯图等。

<a id="integration-change"></a>
### 6. 实施整体变更控制

定义：
审查所有变更请求，批准变更，管理对可交付成果、项目文件和项目管理计划的变更，并对处理结果进行沟通的过程。

作用：
确保已记录在案的变更得到综合评审，避免局部改动放大整体项目风险。

主要输入：

- 项目管理计划
- 项目文件
- 工作绩效报告
- 变更请求
- 事业环境因素
- 组织过程资产

工具与技术：

- 专家判断
- 变更控制工具
- 数据分析
- 决策
- 会议

主要输出：

- 批准的变更请求
- 变更日志
- 项目管理计划更新
- 项目文件更新

高频原文点：

- 配置控制关注可交付成果和技术规范。
- 变更控制关注项目文件、可交付成果和基准的识别、记录、批准与否决。
- 配置管理活动包括：识别配置项、记录并报告状态、核实与审计。
- 变更管理活动包括：识别变更、记录变更、做出决定、跟踪变更。

<a id="integration-close"></a>
### 7. 结束项目或阶段

定义：
终结项目、阶段或合同的所有活动的过程。

作用：

- 存档项目或阶段信息，完成计划工作
- 释放组织团队资源，展开新的工作

主要输入：

- 项目章程
- 项目管理计划
- 项目文件
- 验收的可交付成果
- 立项管理文件、协议、采购文档、组织过程资产

工具与技术：

- 专家判断
- 数据分析：文件分析、回归分析、趋势分析、偏差分析
- 会议

主要输出：

- 最终产品、服务或成果
- 项目最终报告
- 组织过程资产更新

高频原文点：

- 如果项目提前终止，结束过程还要调查并记录提前终止原因。
- 验收的可交付成果可以是最终交付物，也可能是阶段性交付物或中间成果。
- 文件分析、回归分析、趋势分析和偏差分析都服务于总结经验、完善测量指标和改进未来项目。

<a id="integration-confusions"></a>
## 四、本章最容易混淆的点

- 项目章程 vs 项目管理计划
- 项目管理计划 vs 项目文件
- 配置控制 vs 变更控制
- 监督 vs 控制
- 批准的变更请求 vs 变更日志

<a id="integration-review"></a>
## 六、复习提醒

整合管理不适合只背七个过程名。更有效的复习顺序是：

1. 先吃透项目章程与项目管理计划。
2. 再吃透工作绩效数据、工作绩效报告和变更请求的流转。
3. 最后把整体变更控制和项目收尾闭环彻底走顺。

这样做，案例题和论文题的答案会更稳。
